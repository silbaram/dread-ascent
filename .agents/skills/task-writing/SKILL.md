# Task Writing

이 스킬은 Planner 역할로 요구사항을 Task로 분해하고 작성할 때 적용됩니다.
독립적이고, 검증 가능하며, 적절한 크기의 Task를 만들기 위한 가이드입니다.

## 핵심 규칙

1. **하나의 Task는 하나의 기능 단위다**: "로그인 + 회원가입 + 비밀번호 찾기"는 3개의 Task다.
2. **수용 조건은 검증 가능해야 한다**: "성능이 좋아야 한다" (X) → "응답 시간 200ms 이내" (O).
3. **Task는 독립적으로 완료 가능해야 한다**: 다른 Task 완료를 기다려야 시작할 수 있으면 의존성을 명시한다.
4. **크기는 Developer가 한 세션에 완료할 수 있어야 한다**: L 이상은 분할을 검토한다.
5. **불확실한 요소는 [TBD]로 표시한다**: 추측하지 말고 명시적으로 미정 상태를 기록한다.

## Task 작성 형식

### 표준 Task 파일 구조

```markdown
# [task-id] Task 제목

## 메타데이터
| 항목 | 값 |
|------|-----|
| 상태 | BACKLOG |
| 우선순위 | P0 / P1 / P2 |
| 크기 | S / M / L |
| 담당 | - |

## 설명

이 Task가 왜 필요한지, 사용자에게 어떤 가치를 제공하는지 간결하게 기술한다.

## 수용 조건

- [ ] 조건 1: 구체적이고 검증 가능한 문장
- [ ] 조건 2: ...
- [ ] 조건 3: ...

## 참고사항

구현에 도움이 되는 맥락 정보, 제약 조건, 관련 문서 링크 등.

## 의존성

- 선행 Task: (없으면 "없음")
- 관련 Task: (참고용)
```

## 패턴 및 예시

### 패턴 1: 수용 조건 작성법

수용 조건은 **"무엇이 되어야 하는가"**를 명확히 한다.

```
// 나쁜 수용 조건 - 모호하고 검증 불가
- [ ] UI가 예뻐야 한다
- [ ] 빠르게 동작해야 한다
- [ ] 에러 처리가 되어야 한다
- [ ] 잘 작동해야 한다

// 좋은 수용 조건 - 구체적이고 검증 가능
- [ ] 로그인 폼에 이메일, 비밀번호 입력 필드가 있다
- [ ] 이메일 형식이 잘못되면 "올바른 이메일을 입력하세요" 메시지가 표시된다
- [ ] 로그인 성공 시 대시보드 페이지로 이동한다
- [ ] 로그인 실패 시 "이메일 또는 비밀번호가 올바르지 않습니다" 메시지가 표시된다
- [ ] 3회 연속 실패 시 30초 동안 로그인 시도가 차단된다
```

### 패턴 2: Task 크기 산정 (S/M/L)

```
S (Small) - 단순 변경, 1~2개 파일 수정
  예: 버튼 텍스트 변경, 환경 변수 추가, 유효성 규칙 1개 추가

M (Medium) - 단일 기능 구현, 3~5개 파일 수정
  예: 로그인 API 구현, 상품 목록 페이지 구현, 알림 발송 기능

L (Large) - 복합 기능 구현, 5개 이상 파일 수정
  예: 결제 프로세스 전체, 검색 시스템 구현
  → L은 M 2~3개로 분할 검토

XL → 반드시 분할한다. 하나의 Task로 작성하지 않는다.
```

### 패턴 3: 기능 분해 전략

큰 기능을 독립적 Task로 나누는 기준.

```
기능: "사용자 인증 시스템"

// 나쁜 분해 - 계층별 분리 (프론트/백 따로)
task-001: 로그인 UI 구현
task-002: 로그인 API 구현
task-003: 로그인 DB 스키마
→ 문제: 각 Task가 독립적으로 완료 검증 불가

// 좋은 분해 - 기능 단위 수직 분리
task-001: 이메일/비밀번호 로그인 (S: UI + API + DB)
task-002: 로그인 실패 처리 (S: 메시지 표시 + 시도 제한)
task-003: 회원가입 (M: 폼 + 가입 처리 + 이메일 인증)
task-004: 비밀번호 재설정 (M: 요청 + 이메일 발송 + 재설정 처리)
task-005: 자동 로그인 유지 (S: 토큰 갱신 + 세션 유지)
→ 각 Task가 독립적으로 구현/검증 가능
```

### 패턴 4: 의존성 관리

```
// 명시적 의존성 - 선행 Task 없이 시작 불가
task-001: 사용자 테이블 생성       ← 선행 필요
task-002: 회원가입 기능            ← task-001에 의존
task-003: 로그인 기능              ← task-001에 의존

// 의존성 기록 방법
## 의존성
- 선행 Task: task-001 (사용자 테이블이 존재해야 함)

// 의존성 최소화 원칙
// 가능하면 의존성 없이 독립 실행되도록 Task를 설계한다.
// 의존성이 많으면 분해 방식을 재검토한다.
```

### 패턴 5: 우선순위 기준

```
P0 (Critical) - 이것 없이는 시스템이 동작하지 않음
  예: 핵심 데이터 모델, 인증 시스템, 메인 기능

P1 (Important) - 필수 기능이지만 핵심 동작 후 구현 가능
  예: 입력 검증, 에러 처리 개선, 보조 기능

P2 (Nice to have) - 있으면 좋지만 없어도 동작
  예: UI 개선, 성능 최적화, 편의 기능

// 스프린트 구성 시
// P0 전체 + P1 일부를 스프린트에 배치한다.
// P0가 없으면 P1부터 배치한다.
```

## 주의사항

- Task에 구현 방법(how)을 지시하지 않는다. 무엇을(what) 달성해야 하는지만 기술한다
- 수용 조건의 개수는 3~7개가 적절하다. 너무 많으면 Task 분할을 검토한다
- 기술 결정(어떤 라이브러리, 어떤 패턴)은 Planner가 아닌 Developer/project.md의 영역이다
- 기존 plan.md의 요구사항과 Task가 일치하는지 추적 가능해야 한다
