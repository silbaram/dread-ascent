# Code Review Checklist

이 스킬은 Reviewer 역할로 코드 리뷰를 수행할 때 적용됩니다.
리뷰 판정의 일관성과 객관성을 확보하기 위한 체크리스트입니다.

## 핵심 규칙

1. **수용 조건 충족이 최우선이다**: Task의 수용 조건을 모두 만족하는지 먼저 확인한다.
2. **project.md가 기술 기준이다**: 코딩 규칙, 기술 스택, 구조가 project.md와 일치해야 한다.
3. **리뷰어는 코드를 수정하지 않는다**: 문제를 발견하고 기록만 한다. 수정은 Developer의 책임이다.
4. **PASS/REJECT 판정은 근거와 함께 기록한다**: 감정이나 취향이 아닌 문서 기준으로 판정한다.
5. **범위 밖 코드는 리뷰하지 않는다**: 현재 Task와 관련 없는 코드의 품질 문제는 별도 Task로 제안한다.

## 리뷰 체크리스트

### 1단계: 수용 조건 검증 (MUST - 하나라도 실패 시 REJECT)

```
[ ] Task 파일의 수용 조건이 모두 구현되었는가
[ ] 수용 조건에서 요구하는 동작이 실제로 작동하는가
[ ] 수용 조건의 범위를 초과하는 구현이 포함되지 않았는가
```

### 2단계: 기술 기준 준수 (MUST - 위반 시 REJECT)

```
[ ] project.md에 명시된 기술 스택만 사용하였는가
[ ] project.md의 코딩 규칙을 따르는가
[ ] project.md의 프로젝트 구조를 유지하는가
[ ] project.md에 없는 외부 의존성이 추가되지 않았는가
```

### 3단계: 코드 품질 (SHOULD - 심각한 경우 REJECT, 경미하면 WARN)

```
[ ] 함수/클래스의 이름이 역할을 명확히 표현하는가
[ ] 함수가 한 가지 역할만 수행하는가
[ ] 중복 코드가 없는가 (3회 이상 반복)
[ ] 매직 넘버/문자열이 상수로 추출되었는가
[ ] 에러 처리가 적절한가 (빈 catch 블록 없음)
[ ] 불필요한 주석이 없는가 (코드로 설명 가능한 주석)
```

### 4단계: 안정성 (SHOULD - 심각한 경우 REJECT)

```
[ ] 입력 검증이 시스템 경계에서 수행되는가
[ ] 널/빈값 처리가 적절한가
[ ] 외부 API 호출 시 타임아웃이 설정되어 있는가
[ ] 비동기 작업의 에러가 처리되는가
[ ] 민감 정보(비밀번호, 토큰)가 로그에 노출되지 않는가
```

### 5단계: 테스트 (SHOULD - project.md 기준에 따라)

```
[ ] 핵심 비즈니스 로직에 테스트가 작성되었는가
[ ] 테스트가 실제 동작을 검증하는가 (구현 세부사항이 아님)
[ ] 경계값/예외 케이스가 테스트되었는가
[ ] 기존 테스트가 깨지지 않았는가
```

## 판정 기준

### PASS
- 1~2단계 전체 통과
- 3~5단계에서 경미한 개선 사항만 있음
- 개선 사항은 리뷰 리포트에 WARN으로 기록

### REJECT
- 1단계 실패: 수용 조건 미충족
- 2단계 실패: project.md 위반
- 3~5단계 심각: 보안 취약점, 데이터 손실 위험, 심각한 구조 문제

### WARN (PASS와 함께)
- 개선하면 좋지만 현재 Task의 수용 조건과 무관한 사항
- 향후 리팩토링 대상으로 기록

## 리뷰 리포트 작성 형식

```
# Review Report: [task-id]

## 판정: PASS / REJECT

## 수용 조건 검증
- [x] 조건 1: 충족 (구현 확인)
- [ ] 조건 2: 미충족 (사유 기술)

## 발견 사항

### REJECT 사유 (해당 시)
1. [파일:라인] 설명 + 근거 문서 (project.md 섹션 등)

### WARN (개선 권고)
1. [파일:라인] 설명 + 개선 방향

## 요약
판정 근거 한 줄 요약
```

## 리뷰 원칙

### 해야 할 것
- 문서(수용 조건, project.md) 기준으로 판단한다
- 문제의 **심각도**를 명시한다 (REJECT 사유 vs WARN)
- 구체적 파일/라인을 지목하여 피드백한다
- 좋은 코드가 있으면 언급한다

### 하지 말아야 할 것
- 개인 취향으로 판정하지 않는다 ("나라면 이렇게 했을 텐데")
- Task 범위 밖 코드의 문제로 REJECT하지 않는다
- 코드를 직접 수정하지 않는다
- 모호한 피드백을 남기지 않는다 ("이상해 보인다" → 구체적 문제와 근거 명시)

## 주의사항

- 리뷰 기준은 project.md > 이 스킬 순이다. project.md에 구체적 기준이 있으면 그것을 따른다
- REJECT 시 Developer가 재작업할 수 있도록 구체적 수정 방향을 제안한다
- 동일한 패턴의 문제가 여러 곳에 있으면 대표 사례 1~2개만 지적하고 "동일 패턴 N건"으로 요약한다
